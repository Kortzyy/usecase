Use Case Diagram — Technical Explanation

01
v6.0 Key Changes

The biggest v6 addition is Section T — Triangulation Pipeline: 4 new use cases (UC-T1 to UC-T4) representing the FastAPI backend process that converts GPS + bearing into estimated fire coordinates before AI is called. UC-A4b is now an «include» of A1 (bearing is always captured, not just sometimes). UC-H3 updated: Claude receives fire_estimated_lat/lng, never raw bearings. UC-F3 is new for CHG#4 hydrant ground-truth authority.

02
Section T — Why a Separate Section?

  • Triangulation is a deterministic backend computation (FastAPI geometry) — not AI, not user-driven
  • It is placed between Section A (citizen input) and Section B (admin operations) because it runs immediately after every report submission before anything else
  • UC-T1 is system-triggered; it has no actor association — the «system» label is shown in the diagram
  • UC-T3 «extends» T1 because centroid computation only happens with 3+ valid bearing reports
  • UC-T4 «includes» T1 because a confidence badge is always generated regardless of method

03
«include» vs «extend» in v6

  • «include» = always triggered: A1→A2 (GPS always), A1→A3 (photo always), A1→A4b (bearing always), T1→T2 (ray intersection always for 2+ reports), T1→T4 (confidence badge always), B1→B1a (map layers always), B6→B6a (BFP report always on resolve), D4→D4a, E3→E3a, F2→F2a
  • «extend» = conditional: A1→A4 (only on SOS hold), T1→T3 (only if 3+ reports), C6→C6a (only if comment added), H2→H4 (only if admin regenerates)

04
RBAC in Use Cases

  • Alarm Authority: UC-D3 (FV requests) vs UC-D4 (BFP executes) are separate use cases — RBAC hard-coded
  • Hydrant Ground-Truth: UC-F3 is only available after UC-F2a approval — acceptance grants the authority
  • Equipment scoping: UC-F1 is shared but scoped — Super Admin sees all; each Sub-Admin type sees only their own category
  • Map updates: UC-E3 + E3a enforces Barangay can only request, not publish directly
  • AI access: Triggering AI summaries (UC-H1, H2) is limited to Admin roles and above

05
Claude API Role (v6)

  • Claude Haiku is used for testing AND deployment (CHG#2); Sonnet only if pilot quality is insufficient
  • Claude receives ONLY post-triangulated data: fire_estimated_lat/lng (never raw bearings), all photos and descriptions, involvement count, incident type
  • Claude outputs: ai_summary, ai_keywords, ai_priority (Low/Med/High), ai_last_updated, structured BFP fire-out report
  • triangulation_confidence is shown separately on dashboards — NOT folded into AI priority; dispatchers interpret it themselves

06
8 Actors Summary (v6)

  • General User — Submit, track, receive notifications (Mobile)
  • Admin Barangay 76 — Full incident lifecycle, GIS web with confidence badges (Web)
  • Response Teams — Receive, navigate to triangulated coords, report status (Mobile)
  • Sub-Admin BFP — EXECUTE alarm, fire codes, BFP mobile dashboard (Mobile)
  • Sub-Admin FV — REQUEST alarm, verify, dispatch, hydrant updates (Mobile)
  • Sub-Admin Brgy/Medic/Police — Dept ops, map requests (Mobile)
  • Super Admin M2STZ — Full system control (Web)
  • Claude API — Post-triangulation AI (Haiku · Sonnet fallback)
